# Технический аналог систем вроде iAorta (ОАС на нативной КТ)

Документ описывает, **как подобные системы устроены инженерно**, и как в **AIVision** сейчас сделан **только демонстрационный заглушечный пайплайн** (без обученной модели и без клинической валидации).

## 1. Продуктовый контекст (iAorta и аналоги)

**Задача:** раннее выявление острого аортального синдрома (ОАС: расслоение, интрамуральная гематома, разрыв и т.д.) на **неконтрастной** КТ, когда классическая CTA недоступна или задерживается.

**Идея обучения «учитель → ученик» (cross-modal distillation):**

- **Учитель** обучается на данных с **высокой контрастностью сосудов** (CTA / артериальная фаза): проще сегментировать аорту и поражения.
- **Ученик** получает на вход только **нативные** тома и учится имитировать:
  - вероятность события (классификация тома или патча);
  - сегментационные маски (согласованные с учителем в общем пространстве координат);
  - вспомогательные карты уверенности / attention.

Так снимают нехватку масштабных размеченных нативных наборов, сохраняя клинически полезный выход.

## 2. Архитектура пайплайна (как в зрелом продукте)

### 2.1. Интеграция (PACS / RIS / viewer)

- **Триггер:** появление нового исследования (HL7 / DICOMweb / локальный watcher).
- **Маршрутизация:** только **CT Chest/Angio**, нужная толщина среза, отсутствие явного motion-artifact (опционально QC-модуль).
- **Асинхронность:** инференс в очереди (GPU worker), результат пишется в БД и доставляется в UI по подписке.

### 2.2. Модуль отбора фазы (phase / contrast classifier)

Небольшая сеть или правило:

- гистограмма HU в ROI средостения;
- доля вокселей выше порога (например, >130 HU) в ожидаемой зоне аорты;
- признаки «артериальной фазы» → если контраст высокий, **нативная модель не запускается** или запускается другая ветка.

Цель — снизить ложные срабатывания и не смешивать домены.

### 2.3. Ядро: 3D CNN (или гибрид CNN + Transformer)

**Вход:** объём HU (часто изотропический ресемплинг или multi-scale 3D crop вокруг аорты).

**Голова A — классификация:** `P(OAS)` или много-класс (IMH / PAU / dissection…).

**Голова B — сегментация:** маски стенки / просвета / тромбозированного ложного канала (зависит от протокола разметки).

**Объяснимость:**

- Grad-CAM / Grad-CAM++ в 3D;
- attention rollout в ViT-ветке;
- отдельная лёгкая сеть saliency, обученная на согласовании с масками учителя.

### 2.4. Обучение дистилляции (упрощённо)

Пары (нативный том `N`, CTA том `C`) после регистрации или слабой геометрической привязки:

- **KL / L2** между логитами учителя и ученика;
- **Dice / CE** между масками учителя (на CTA, спроецированными) и предсказанием ученика на `N`;
- веса на редкие классы и hard-negative mining.

Регуляризация: adversarial noise, domain randomization (разные сканеры, ядра, шум).

### 2.5. Модуль оповещения в UI

- порог по `P` и политика больницы (например, только «высокий риск»);
- поп-ап с **кратким текстом**, **ссылкой на ключевой срез**, **оверлеем маски** и **картой активации**;
- аудит: кто подтвердил / отклонил, время до прочтения.

## 3. Что именно реализовано в AIVision сейчас

Файл `frontend/src/lib/ctPathologyScreen.ts` — **много-классовый КТ-скрининг по порогам HU и зонам** (без обученной нейросети):

- сканирование **всех загруженных срезов**; агрегация находок по серии;
- классы-эвристики: гиперденсивные очаги, лёгочные зоны пониженной / повышенной плотности, кальцинатная нагрузка; учёт эвристики контрастной фазы;
- цветная маска на **текущем аксиале** + тултипы по классу; модальное окно при высокой «уверенности» эвристики.

Чувствительность/специфичность **не валидированы**; это вспомогательный инструмент, не заключение радиолога.

## 4. Путь к «настоящему» аналогу iAorta в этом репозитории

1. Экспорт изометрического подтома или full volume в `Float32Array` + метаданные (уже близко к `ctVolume` / vtk пути).
2. ONNX / TensorFlow.js / отдельный inference server (Python + Triton) с GPU.
3. Отдельные веса для phase classifier и 3D backbone.
4. Хранение результатов (JSON + UID исследования) и повторное открытие в viewer.
5. Клинический протокол валидации и регистрация ПО — вне scope кода.
