# Техническое задание

## 1. Название проекта

`AIVision` — локальный диагностический инструмент для радиолога с поддержкой DICOM, КТ-серий, MPR, 3D-визуализации и подготовкой к сегментации.

## 2. Цель

Создать приложение, которым радиолог сможет пользоваться для:
- открытия локальных DICOM-файлов и серий;
- просмотра аксиальных срезов КТ;
- переключения между срезами;
- базовых измерений;
- MPR-реконструкции;
- 3D-визуализации костей, суставов и сосудистых структур;
- дальнейшего расширения до сегментации и экспорта результатов.

## 3. Важное ограничение

На первом этапе создается не сертифицированное медицинское изделие, а инженерный продукт и прототип рабочего места радиолога.

Это означает:
- приложение нельзя позиционировать как готовую медицинскую систему для клинических решений без отдельной валидации;
- на первом этапе делаем упор на качество интерфейса, корректность отображения DICOM и удобство врача;
- юридические и регуляторные требования выносятся в отдельный этап после MVP.

## 4. Формат поставки

Первичная поставка:
- локальное desktop/web-приложение для Windows;
- интерфейс на русском языке;
- запуск на рабочей станции врача без сложной установки;
- работа прежде всего с локальными DICOM-исследованиями.

## 5. Целевая аудитория

- врач-рентгенолог;
- врач функциональной диагностики;
- хирург или травматолог, которому нужны 3D-модели;
- клиника или частная лаборатория.

## 6. Основные пользовательские сценарии

### 6.1 Загрузка исследования

Пользователь должен иметь возможность:
- выбрать папку с DICOM-файлами;
- увидеть список исследований и серий;
- открыть нужную серию КТ;
- видеть базовую служебную информацию о серии.

### 6.2 Просмотр серии

Пользователь должен иметь возможность:
- листать срезы колесом мыши и ползунком;
- менять window/level;
- приближать и перемещать изображение;
- инвертировать изображение;
- сбрасывать вид к исходному состоянию.

### 6.3 Измерения

Пользователь должен иметь возможность:
- измерять расстояние;
- измерять угол;
- ставить метки и аннотации;
- видеть значения в мм и градусах.

### 6.4 MPR

Пользователь должен иметь возможность:
- открыть аксиальную, сагиттальную и корональную плоскости одновременно;
- синхронно перемещаться по объему;
- видеть перекрестие и взаимное положение срезов.

### 6.5 3D-визуализация

Пользователь должен иметь возможность:
- построить объемную 3D-модель из серии;
- вращать модель;
- управлять прозрачностью;
- выбирать пресеты визуализации: кость, мягкие ткани, сосудистая маска;
- быстро возвращаться к стандартному виду.

### 6.6 Подготовка к сегментации

Пользователь должен иметь возможность:
- выбрать область интереса;
- применить порог по плотности;
- получать черновую маску костей, суставов или сосудов;
- в будущем редактировать маску вручную.

## 7. Функциональные требования

## 7.1 Модуль импорта

Система должна:
- импортировать локальные DICOM-файлы;
- группировать файлы по исследованию, серии и экземплярам;
- читать основные DICOM-теги;
- корректно обрабатывать порядок срезов.

Минимальные теги первого этапа:
- PatientName;
- PatientID;
- StudyInstanceUID;
- SeriesInstanceUID;
- SOPInstanceUID;
- Modality;
- StudyDate;
- SeriesDescription;
- InstanceNumber;
- ImagePositionPatient;
- ImageOrientationPatient;
- PixelSpacing;
- SliceThickness;
- RescaleSlope;
- RescaleIntercept;
- WindowCenter;
- WindowWidth.

## 7.2 Модуль 2D-просмотра

Система должна:
- отображать отдельный срез;
- отображать серию срезов как стек;
- поддерживать wheel-scroll;
- поддерживать оконные пресеты;
- поддерживать измерительные инструменты.

## 7.3 Модуль MPR

Система должна:
- собирать объем из серии;
- строить аксиальный, сагиттальный и корональный вид;
- синхронизировать курсор между окнами;
- обеспечивать приемлемую производительность на типовых КТ-сериях.

## 7.4 Модуль 3D

Система должна:
- выполнять volume rendering;
- отображать 3D-модель в реальном времени;
- поддерживать базовые пресеты opacity/transfer function;
- иметь стартовые пресеты для костной ткани и мягких тканей;
- закладывать архитектуру под будущую визуализацию сосудов.

## 7.5 Модуль сегментации

Первая версия должна:
- поддерживать threshold segmentation;
- поддерживать region growing как опциональный режим следующего этапа;
- сохранять маску в памяти сеанса.

## 7.6 Экспорт

Не входит в первый MVP, но архитектура должна учитывать:
- экспорт скриншотов;
- экспорт STL/OBJ для 3D-моделей;
- экспорт масок и вторичных данных.

## 8. Нефункциональные требования

### 8.1 Производительность

- открытие стандартной серии КТ до 500-1500 срезов без критических зависаний;
- отображение первого среза не более чем через несколько секунд после загрузки;
- плавная навигация по стеку на типовой рабочей станции.

### 8.2 Платформа

- приоритет: Windows 10/11;
- интерфейс должен работать локально;
- архитектура должна позволять позже упаковать продукт как desktop-приложение.

### 8.3 Язык интерфейса

- основной язык: русский;
- все ключевые кнопки и действия должны быть понятны врачу без технической подготовки.

### 8.4 Безопасность

- локальные DICOM-данные не отправляются во внешние сервисы без явного действия пользователя;
- в логах не хранить чувствительные медицинские данные в открытом виде;
- предусмотреть анонимизацию как отдельный этап.

## 9. Рекомендуемая архитектура

Для первого этапа принимаем следующую архитектуру:

- `Frontend`: React + TypeScript + Vite;
- `2D/MPR`: Cornerstone3D;
- `3D`: vtk.js;
- `DICOM parsing/loading`: dicom-parser и DICOM image loader;
- `State`: Zustand или Redux Toolkit;
- `UI`: собственный минималистичный интерфейс без тяжелой медицинской ERP-логики.

Причина выбора:
- это наиболее реалистичный стек для постепенной разработки в Codex;
- можно начать с 2D и без переписывания перейти к MPR/3D;
- стек совместим с локальной разработкой и прототипированием.

## 10. Ограничения проекта

- пользователь проекта не является разработчиком;
- поэтому все шаги разработки должны быть максимально атомарными;
- каждая стадия должна завершаться проверяемым результатом;
- приоритет на ранних этапах: стабильность и понятность, а не максимальный набор функций.

## 11. Этапы реализации

### Этап 1. Основа проекта

Результат:
- создан проект;
- запускается локально;
- есть стартовый интерфейс.

### Этап 2. Импорт DICOM

Результат:
- можно выбрать локальные DICOM-файлы;
- видно список серий.

### Этап 3. 2D Viewer MVP

Результат:
- можно открыть серию;
- листать срезы;
- менять zoom, pan, window/level.

### Этап 4. Измерения и аннотации

Результат:
- линейка, угол, метки.

### Этап 5. MPR

Результат:
- трехоконный просмотр объема.

### Этап 6. 3D Rendering

Результат:
- объемная модель;
- пресеты плотностей.

### Этап 7. Черновая сегментация

Результат:
- пороговое выделение кости или сосудов.

## 12. Критерии приемки MVP

MVP считается готовым, если:
- приложение запускается локально на Windows;
- пользователь открывает папку DICOM;
- система корректно показывает серию КТ;
- доступна навигация по срезам;
- работают zoom, pan, window/level;
- доступны базовые измерения;
- нет критических падений на типовом исследовании.

## 13. Что не входит в MVP

- медицинская сертификация;
- PACS/RIS интеграция;
- полноценный облачный режим;
- автоматическая ИИ-диагностика;
- идеальная сегментация сосудов и суставов на первом этапе;
- совместная работа нескольких врачей.
